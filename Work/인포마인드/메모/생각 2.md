## 생각

<br>

- use client 를 쓰는 SPA 형태는 보안이 너무 구림 → 이런 형태는 사용자의 직접적인 api호출을 막을 도구가 부족함 (httponly cookie 사용불가, 어느 IP로 올지 모르니 cors, ip필터 사용제한 등등), 오는 거 각각 권한 검증도 해야 함(프론트와 중첩됨, 범위가 너무 넓음) → CMS페이지는 아마 내부 네트워크일 거고, 관리자 계정을 제외하고는 막을거니까 상관이 없을 수도 있음(가능성)
    - 현재 개발자 도구에서 document.cookie 치면 access token이 노출됨 (XSS 공격, httponly cookie면 노출이 안되는데 클라이언트 단에서도 개발자가 사용할 수 없음, refresh token은 현재 httponly cookie라 노출안됨)
- 적어도 사용자 페이지는 위와 같은 패턴은 최대한 지양해야 함 → BFF 패턴으로 가야 함. 클라이언트에서 api 호출이 필요하다면 반드시 Next.js 서버를 거쳐서 호출해야 한다. → 그래야 CORS, IP 화이트리스트로 갈 수 있음 (Next.js IP만 뚫어주면 됨) 혹은 인프라적으로 Next 서버 호출만 불러들일 수 있음
- 결론적으로는 API서버의 직접적인 호출을 막아서, 프론트 쪽에서만 권한 검증에 집중할 수 있는 형태가 되어야 함

<br>

- 현재 access token 무한 증식할 수 있는 형태 → 이거 막으려면 DB에 access token 몇 개 만들었는지 저장해야 함 → 토큰 만료 시간도 있어서 막 필요한 것도 아니고,  그리고 이전에도 세션 무한증식함 → 필요하지 않을 수 있음
- JWT 로그아웃 시 RefreshToken을 삭제해도 AccessToken은 동작됨 → 이거 막으려면 로그아웃 때 받은 Access Token을 만료 기간 동안 저장했다가 막아야 함 (구현 시 Redis가 편함) → 만료기간도 있어서 반드시 필요하지 않을 수 있음

<br>

- 로그인 유지 구현 → 그냥 로그인 시 access token만 주고, 로그인 유지 시 refresh token을 같이 주면 됨
